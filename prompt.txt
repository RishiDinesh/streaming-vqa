Paper story and experiment strategy
-----------------------------------
- We care about both streaming datasets:
  - `RVS-Ego`
  - `RVS-Movie`
- Main intended claim:
  - `DuoAttention` alone is not expected to beat streaming SOTA.
  - The useful claim is that `DuoAttention` can be added on top of `ReKV` to reduce memory while preserving most of `ReKV`'s accuracy.
- Because compute is limited, the plan is:
  1. run subsamples first
  2. compare `duo_plus_rekv` at `s=0.375`, `s=0.5`, and `s=0.75`
  3. choose exactly one hybrid sparsity for full eval
  4. run the full datasets only once with the promoted set
- We do not plan to spend compute on `duo_streaming (s=0.0)` unless later analysis shows it is necessary.

Core streaming semantics
------------------------
- causal frame ingest only
- one sampled frame per forward pass
- shared state across questions in the same video
- no offline full-video prefill
- no LongBench-style tail replay
- question-time availability is based on sampled timestamps strictly earlier than `end_time`
- ReKV and DuoAttention must use the same sampled-frame schedule

Important fixes already made
----------------------------
10. ROCm 7 / transformers 4.55 performance and compatibility (2026-04-10)
- Launcher scripts now prefer `/opt/venv` (torch 2.9 + ROCm 7.0) over the `duo`
  conda env (torch 2.4 + ROCm 6.1); ROCm 7 gives 3–5× speedup on MI300X gfx942.
- Measured: vision encoder 18.7→8.1 ms/frame, decode 60.8→11.8 ms/token.
- `torch.compile(mode='reduce-overhead')` and `torch.set_float32_matmul_precision('high')`
  applied at startup in `run_eval.py`; auto-enabled on ROCm 7+; disable with `PYTORCH_COMPILE=0`.
- transformers 4.55: `model.language_model` is now `Qwen2Model` directly (no `.model` wrapper).
  Fixed in `duo_attn/patch/llava_onevision.py` (`_get_qwen2_layers`, `get/set/map_qwen2_full_attention_heads`)
  and `streaming/ReKV/methods.py` (`DuoPlusReKVStreamingMethod.__init__`).
- decord threaded-decoder crash: this build of decord crashes whenever `num_threads >= 4`.
  Fix is a one-line clamp in `_sample_decord_video`: `safe_threads = min(max(decode_threads, 1), 2)`.
  `num_threads=2` is ~30% faster than 1 and never crashes. `VIDEO_DECODE_THREADS` can stay at 4
  in the scripts — it just gets silently clamped inside the reader. `get_frame`/`get_frames`
  need no fallback logic and are kept simple.

Monitoring and resume behavior
------------------------------
Progress bars are implemented for the long-running steps:
- streaming eval:
  - outer bar over videos
  - inner bar over sampled frames for the current video
- feature precompute:
  - outer bar over videos
  - inner bar over sampled frames / batches for the current video
- profiling:
  - frame bar
  - probe bar
- judging already has a conversation progress bar

Resume behavior:
- full and subsample eval write checkpoints to the final output JSON path
- use `RESUME=1` in the shell launchers to continue interrupted runs
- output JSON now carries:
  - `run_state`
  - `videos`
  - optional `in_progress_video`
- if the run stops after some conversations inside a video, resume will reconstruct the stream state by replaying frames and continue from the next unanswered conversation
- the subsample matrix runner also keeps:
  - `outputs/evaluations_streaming/subsample_runner/status.txt`
  - per-step logs under `outputs/evaluations_streaming/subsample_runner/logs/`

Recommended knobs for single-GPU runs
-------------------------------------
- keep `USE_FEATURE_CACHE=1`
- keep `CLEAR_CUDA_CACHE_ON_RESET=0`
- start with `VIDEO_DECODE_THREADS=4`
- keep `FLUSH_EVERY_VIDEOS=1`
- keep `FLUSH_EVERY_CONVERSATIONS=1`
- run only one measured method on the GPU at a time

What to avoid
-------------
- batching multiple streamed frames into one LM ingest step
- answering multiple questions from the same video in parallel
- running multiple measured methods concurrently on the same GPU for final latency/memory numbers
- reintroducing an unsafe batched feature-extraction shortcut
