Streaming ReKV vs DuoAttention handoff
======================================

Use this file as the starting context for a new chat.

Current goal
------------
- This repo work is only for the streaming `ReKV` vs `DuoAttention` lane.
- Hardware/runtime assumption:
  - one local AMD `MI300X`
  - no SLURM for this task
- Do not push to GitHub. Only the user will push.

Core streaming semantics
------------------------
- causal frame ingest only
- one sampled frame per forward pass
- shared state across questions in the same video
- no offline full-video prefill
- no LongBench-style tail replay
- ReKV and DuoAttention must use the same sampled-frame schedule

Validated methods
-----------------
- `full_streaming`
- `duo_streaming`
- `rekv`
- `duo_plus_rekv`

Current promoted settings
-------------------------
- `duo_streaming`
  - `sparsity = 0.5`
- `rekv`
  - `retrieve_size = 64`
  - `n_local = 15000`
- `duo_plus_rekv`
  - `sparsity = 0.375`
  - default deploy recent size
  - `retrieve_size = 64`
  - `n_local = 15000`

What was fully validated
------------------------
Official subsample-only slices:
- `RVS-Ego subsample5`

Main findings
-------------
- `rekv` is the most consistently strong standalone method across the validated subsamples.
- `duo_streaming (s=0.5)` is a meaningful memory-saving streaming baseline against `full_streaming`, but its quality-latency tradeoff depends on the dataset.
- `duo_plus_rekv (s=0.375)` is a valid hybrid, but not yet a universal improvement over plain `rekv`.

Best current story
------------------
- `duo_streaming` vs `full_streaming`
- `rekv` as the strongest standalone streaming memory baseline
- `duo_plus_rekv` as a promising hybrid, but not yet a universal winner

Optimized full-eval workflow
----------------------------
1. Precompute shared visual features once:
- `DATASET=rvs_ego bash scripts/run_streaming_full_eval_local.sh precompute`
- `DATASET=rvs_movie bash scripts/run_streaming_full_eval_local.sh precompute`

2. Run the official full package from cache:
- `DATASET=rvs_ego bash scripts/run_streaming_full_eval_local.sh all`
- `DATASET=rvs_movie bash scripts/run_streaming_full_eval_local.sh all`

3. Optional ablation:
- `DATASET=rvs_ego bash scripts/run_streaming_full_eval_local.sh all_with_control`

4. Resume if interrupted:
- `DATASET=rvs_ego RESUME=1 bash scripts/run_streaming_full_eval_local.sh all`
- `DATASET=rvs_movie RESUME=1 bash scripts/run_streaming_full_eval_local.sh all`

5. Judge only:
- `DATASET=rvs_ego bash scripts/run_streaming_full_eval_local.sh judge`
- `DATASET=rvs_movie bash scripts/run_streaming_full_eval_local.sh judge`

Feature-cache validation result
-------------------------------
Validated on one real `RVS-Movie` clip (`tt0090756`) with one conversation:
- raw vs cached matched exactly for:
  - `full_streaming`
  - `duo_streaming (s=0.5)`
  - `rekv`
  - `duo_plus_rekv (s=0.375)`
- ReKV retrieval stats also matched exactly.

Short-clip wall times:
- `full_streaming`: `7.63s -> 6.96s`
- `duo_streaming`: `7.92s -> 7.44s`
- `rekv`: `8.41s -> 7.85s`
- `duo_plus_rekv`: `8.39s -> 7.86s`

Important caveats
-----------------
- Full-dataset eval has not been run end to end yet.
- The local judge is useful, but coarse.
- Do not reintroduce a batched vision-forward cache shortcut unless it is revalidated carefully.
