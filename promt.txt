=============================================

Use this file as the starting context for a new chat session.

What the original task was
--------------------------
- Understand this repo and focus only on the streaming ReKV vs DuoAttention lane.
- Hardware/runtime constraint:
  - local AMD MI300X only
  - no SLURM for this task
- Branch context:
  - work was done locally in this repo
  - user does not want any GitHub pushes by the assistant
- Research goal:
  - compare streaming `DuoAttention`, `ReKV`, and their combination on streaming VideoQA
  - keep evaluation causal and faithful to streaming, not offline prefill
- Important semantic decision already made:
  - in streaming, frames are ingested causally as they arrive
  - no offline full-video prefill
  - no LongBench-style tail replay in the main streaming path
  - ReKV and DuoAttention must use the same frame-ingest schedule

Important fixes that were necessary
-----------------------------------
1. DuoAttention ROCm correctness fix
- File: `duo_attn/patch/attn_compat.py`
- Problem:
  - the SDPA fallback on ROCm used the wrong causal alignment for cached incremental attention when `q_len < k_len`
- Fix:
  - use the correct bottom-right causal mask
- Result:
  - fixed degenerate Duo streaming behavior on MI300X

2. ReKV + DuoAttention combined path on ROCm
- File: `streaming/ReKV/rekv_core/attention/rekv_attention.py`
- Problems:
  - PyTorch SDPA on this stack does not support `enable_gqa`
  - combined path needed to preserve native ReKV retrieval semantics
- Fixes:
  - manually expand KV heads instead of relying on `enable_gqa`
  - disable Duo during `ContextManager.to_retrieve`
  - apply Duo only after retrieval during answer-time LM attention
- Result:
  - `duo_plus_rekv` now means:
    - retrieval policy = native ReKV
    - answer attention policy = DuoAttention post-retrieval

3. Streaming causality fix
- File: `streaming/ReKV/run_eval.py`
- Problem:
  - runner previously ingested one extra sampled frame at question time
- Fix:
  - use upstream-style exclusive cutoff based on `int(end_time * sample_fps)`

Current best method settings
----------------------------
- `duo_streaming`
  - learned mask dir from:
    - `outputs/train/0p5b_sink512_recent1024_maxlen32000_frames64_depth0p1-0p8_needles5_20260328_170632`
  - promoted setting:
    - `sparsity = 0.5`
- `rekv`
  - `retrieve_size = 64`
  - `n_local = 15000`
- `duo_plus_rekv`
  - `sparsity = 0.375`
  - default deploy recent size
  - `retrieve_size = 64`
  - `n_local = 15000`

Main findings
-------------
- `rekv` is the most consistently strong baseline across the completed subsample slices.
- `duo_streaming (s=0.5)` is a real streaming baseline and often saves memory relative to `full_streaming`, but its quality-latency tradeoff is dataset-dependent.
- `duo_plus_rekv (s=0.375)` is technically valid and sometimes complementary, but it is not yet a universal improvement over plain `rekv`.

Best current story:
- `duo_streaming` vs `full_streaming`
- `rekv` as strongest standalone streaming memory baseline
- `duo_plus_rekv` as a promising hybrid, but not yet a universal winner
