WHAT THIS PROJECT DOES
-----------------------
Evaluate 4 streaming video-QA methods on RVS benchmark (ego + movie videos).
Each method ingests one sampled frame at a time and answers open-ended questions
about what it saw — no offline whole-video prefill.
Goal: compare quality (ROUGE-L F1), answer latency, and GPU/CPU memory.


FOUR METHODS
-------------
full_streaming   — plain causal KV cache; all frames kept on GPU
                   flash_attention_2 keeps attention O(seq_len) — no kernel OOM
                   KV cache grows to ~9 GB (movie) / ~13 GB (ego 60-min)

duo_streaming    — Duo head routing: ~25% full-attn heads (all tokens),
                   ~75% streaming heads (sink + recent window only)
                   requires block_sparse_attn kernel
                   window: sink=256 recent=512 (deploy override; trained=512/1024)

rekv             — local window on GPU + CPU offload + top-k cosine retrieval at Q time
                   ~3.2 GB GPU + ~2.3 GB CPU (movie) / ~3.1 GB GPU + ~4 GB CPU (ego)

duo_plus_rekv    — ReKV retrieval then Duo head routing over assembled context
                   ~3.2 GB GPU + ~2.3 GB CPU; approximate hybrid, not paper reproduction

DATA FLOW
----------
datasets.py         load RVS annotation JSON, sort convs by end_time, sample
                    frames at sample_fps, resolve video path (local → HF download)
  ↓
[optional] precompute_features.py → feature_cache.py
                    pre-extract visual tokens; guarantees bit-identical features
  ↓
methods.py          per-video reset; causal ingest (one frame per forward pass)
                    full_streaming:   flash_attention_2 KV cache
                    duo_streaming:    block_sparse_attn for streaming heads
                    rekv:             patch_hf → rekv_attention_forward → ContextManager
                    duo_plus_rekv:    same + Duo weights → rekv_duo_enabled=True
  ↓
answer_question → greedy_decode_with_cache
                    retrieval triggered here for rekv/duo_plus_rekv
                    returned past_key_values = [retrieved blocks] + [question tokens]
  ↓
run_eval.py         write result JSON, checkpoint/resume, sharding

RUNNING EVALUATIONS
--------------------
# Smoke test: 1 video, all 4 methods
bash scripts/run_streaming_subset3_slurm.sh --max-videos 1

# Full eval — rvs-movie (4 chunks, ~5-6 videos each)
for METHOD in full_streaming duo_streaming rekv duo_plus_rekv; do
  for C in 0 1 2 3; do
    DATASET=rvs_movie METHOD=${METHOD} NUM_CHUNKS=4 SPARSITY=0.75 \
    DEPLOY_SINK_SIZE=256 DEPLOY_RECENT_SIZE=512 \
    OUTPUT_ROOT="${PWD}/outputs/evaluations_streaming/rvs-movie/full_eval/run1" \
    sbatch --array=${C} --output="${PWD}/logs/movie-${METHOD}-c${C}-%j.out" \
           scripts/run_streaming_eval_slurm_array.sh
  done
done

# Full eval — rvs-ego (MUST use 10 chunks + 120G; 60-min videos OOM at 4 chunks/64G)
for METHOD in full_streaming duo_streaming rekv duo_plus_rekv; do
  for C in $(seq 0 9); do
    DATASET=rvs_ego METHOD=${METHOD} NUM_CHUNKS=10 SPARSITY=0.75 \
    DEPLOY_SINK_SIZE=256 DEPLOY_RECENT_SIZE=512 \
    OUTPUT_ROOT="${PWD}/outputs/evaluations_streaming/rvs-ego/full_eval/run2" \
    sbatch --array=${C} --mem=120G --output="${PWD}/logs/ego-${METHOD}-c${C}-%j.out" \
           scripts/run_streaming_eval_slurm_array.sh
  done
done

# After all chunks finish — merge then plot
MERGED=outputs/evaluations_streaming/rvs-movie/full_eval/merged
python -m streaming.ReKV.plot_results \
    ${MERGED}/full_streaming.json ${MERGED}/duo_streaming.json \
    ${MERGED}/rekv.json ${MERGED}/duo_plus_rekv.json \
    --output-dir outputs/evaluations_streaming/rvs-movie/full_eval/plots/
python -m streaming.ReKV.compare_subsamples \
    ${MERGED}/full_streaming.json ${MERGED}/duo_streaming.json \
    ${MERGED}/rekv.json ${MERGED}/duo_plus_rekv.json \
    --output-dir outputs/evaluations_streaming/rvs-movie/full_eval/comparison/

COMMON ISSUES AND FIXES
------------------------
avcodec_send_packet error (decord crash)
  Fix: always --video-decode-threads 1 (default in submit scripts; do not increase)

rvs-ego OOM (all methods, not just full_streaming)
  Cause 1: --mem=64G too low; nodes have 125 GB — use --mem=120G
  Cause 2: NUM_CHUNKS=4 → 2-3 videos/job; 60-min KV caches accumulate across videos
  Fix: NUM_CHUNKS=10 (1 video/job) + --mem=120G — cache fully resets between videos

Disk quota exceeded on /u/navdeep during setup
  Cause: CUDA packages (~8 GB) land in ~/.conda/pkgs by default
  Fix: setup.sh redirects via CONDA_PKGS_DIRS=<repo>/.conda_pkgs
       Clean home dir: conda clean --packages --tarballs

CondaToSNonInteractiveError (Terms of Service not accepted)
  Fix: run conda tos accept ... from head node before any batch conda ops

user env retrieval failed requeued held
  Cause: --export=NONE or --export=ONLY_VAR=... blocks SLURM prologue
  Fix: use default --export=ALL; hardcode env vars inside run_eval.sh

Batch job submission failed: requested node config not available
  Fix: use --gres=gpu:rtx_a6000:1 --partition=gpunodes (already in submit scripts)

full_streaming old OOM (FIXED)
  Cause: attn_implementation="eager" materialises O(seq_len²) attention matrix;
         at 108k tokens this was >300 GB regardless of node RAM
  Fix: all methods now use flash_attention_2 (falls back to sdpa)

rekv/duo_plus_rekv generating 1-word answers ("The person<|im_end|>")
  Cause: after retrieval, past_key_values was missing question tokens; decoder
         had no question context and emitted EOS immediately
  Fix (applied in rekv_attention.py ~line 225):
    current_key_value = (h_k, h_v)   # h_k = cat([retrieved, question])
    NOT: current_key_value = (past_k, past_v)  # this was the bug

chunk JSON has videos=[] and null aggregate_metrics
  Cause: job OOM-killed before completing first video
  Fix: delete the empty chunk file and resubmit with NUM_CHUNKS=10 + --mem=120G

BASH_SOURCE[0] wrong path in SLURM
  Cause: SLURM copies scripts to /var/spool/slurmd/jobXXX/slurm_script
  Fix: run_eval.sh hardcodes ROOT=<absolute path>


LIMITATIONS
------------
- full_streaming: GPU KV cache grows to ~9 GB (movie) / ~13 GB (ego); fits on A6000 (48 GB)
  but requires --mem=120G + 1 video/job on rvs-ego due to host RAM accumulation
- duo_plus_rekv is approximate (streaming heads use n_local, not Duo's recent_size)
- decord must use 1 thread on this cluster (avcodec thread safety issue)
- rekv positional encoding: retrieved blocks lose absolute position (distance scaling only)
- sacct unavailable (DB connection refused); use log files to diagnose failed jobs
- Duo sink/recent is a knob: trained=512/1024, deploy override via --deploy-sink-size/recent-size


KEY PARAMETERS
---------------
--method           required   full_streaming | duo_streaming | rekv | duo_plus_rekv
--dataset          rvs_ego    rvs_ego | rvs_movie
--sample-fps       0.5        frames/sec to sample (must match across compared runs)
--max-new-tokens   64         tokens to generate per answer
--n-local          15000      ReKV GPU local window (tokens)
--retrieve-size    64         ReKV top-k blocks from CPU at question time
--n-frame-tokens   196        visual tokens per frame (LLaVA-OV = 196, must match model)
--attn-dir         see scripts path to Duo trained head weights (.tsv files)
--sparsity         0.75       fraction of streaming heads (0=all full, 1=all streaming)
--deploy-sink-size  (trained)  override Duo sink window; trained default=512
--deploy-recent-size (trained) override Duo recent window; trained default=1024
--video-decode-threads 1      keep at 1 (threading crashes on this cluster)
--max-videos       all        limit videos (use 1 for smoke test)
--num-chunks / --chunk-index  sharding; use NUM_CHUNKS=10 for rvs-ego
--resume                      resume from partial result JSON
