WHAT THIS PROJECT DOES
-----------------------
Evaluate 4 streaming video-QA methods on RVS benchmark (ego + movie videos).
Each method ingests one sampled frame at a time and answers open-ended questions
about what it saw — no offline whole-video prefill.
Goal: compare quality (ROUGE-L F1), answer latency, and GPU/CPU memory.


FOUR METHODS
-------------
full_streaming   — plain causal KV cache; keeps all frames on GPU
                   KV cache ~4.3 GB for 60-min video; flash_attention_2 keeps it feasible

duo_streaming    — Duo head routing: full KV for retrieval heads, sink+recent window
                   for streaming heads; requires block_sparse_attn kernel

rekv             — local window on GPU + CPU offload + top-k cosine retrieval at Q time
                   ~3.4 GB GPU + ~4.3 GB CPU for 1800-frame video

duo_plus_rekv    — ReKV retrieval then Duo head routing over assembled context (approximate)
                   ~3.4 GB GPU + ~4.3 GB CPU; not a literal paper reproduction


CURRENT SMOKE-TEST RESULTS (1 video, 259 questions, RVS-Ego, subset1)
----------------------------------------------------------------------
Method                ROUGE-L F1   Latency(s)   Peak GPU    CPU offload
duo_streaming          0.146        0.464        8.4 GB      —
rekv                   0.275        0.082        3.4 GB      4.3 GB
duo_plus_rekv          0.268        0.149        3.4 GB      4.3 GB
full_streaming         OOM (A6000 node; 125 GB RAM exceeded)

duo_streaming scores lower because 50% of heads (s=0.5) use only a small
recent window and miss mid-video context. rekv retrieves relevant blocks
from CPU and wins on quality. Tuning --sparsity lower improves Duo quality
but uses more GPU memory.


DATA FLOW
----------
datasets.py         load RVS annotation JSON, sort convs by end_time, sample
                    frames at sample_fps, resolve video path (local → HF download)
  ↓
[optional] precompute_features.py → feature_cache.py
                    pre-extract visual tokens; guarantees bit-identical features
  ↓
methods.py          per-video reset; causal ingest (one frame per forward pass)
                    full_streaming:   eager KV cache
                    duo_streaming:    blocksparse streaming heads
                    rekv:             patch_hf → rekv_attention_forward → ContextManager
                    duo_plus_rekv:    same + Duo weights → rekv_duo_enabled=True
  ↓
answer_question → greedy_decode_with_cache
                    retrieval triggered here for rekv/duo_plus_rekv
                    returned past_key_values = [retrieved blocks] + [question tokens]
  ↓
run_eval.py         write result JSON, checkpoint/resume, sharding

COMMON ISSUES AND FIXES
------------------------
avcodec_send_packet error (decord crash)
  Fix: always --video-decode-threads 1 (default in submit scripts; do not increase)

Disk quota exceeded on /u/navdeep during setup
  Cause: CUDA packages (~8 GB) land in ~/.conda/pkgs by default
  Fix: setup.sh redirects via CONDA_PKGS_DIRS=<repo>/.conda_pkgs
       Clean home dir: conda clean --packages --tarballs

CondaToSNonInteractiveError (Terms of Service not accepted)
  Fix: run conda tos accept ... from login node before any batch conda ops

user env retrieval failed requeued held
  Cause: --export=NONE or --export=ONLY_VAR=... blocks SLURM prologue
  Fix: use default --export=ALL; hardcode env vars inside run_eval.sh

Batch job submission failed: requested node config not available
  Fix: use --gres=gpu:rtx_a6000:1 --partition=gpunodes (already in submit scripts)

full_streaming OOM (old issue — FIXED)
  Cause: attn_implementation="eager" materialises O(seq_len²) attention matrix;
         at 108k tokens this was >300 GB regardless of node RAM
  Fix: all methods now use flash_attention_2 (falls back to sdpa). KV cache for
       1800-frame video is only ~4.3 GB. OOM is gone, comparability preserved
       (same kernel for all methods; only KV layout differs).

rekv/duo_plus_rekv generating 1-word answers ("The person<|im_end|>")
  Cause: after retrieval, past_key_values was missing question tokens; decoder
         had no question context and emitted EOS immediately
  Fix (already applied in rekv_attention.py line ~225):
    current_key_value = (h_k, h_v)   # h_k = cat([retrieved, question])
    NOT: current_key_value = (past_k, past_v)  # this was the bug

BASH_SOURCE[0] wrong path in SLURM
  Cause: SLURM copies scripts to /var/spool/slurmd/jobXXX/slurm_script
  Fix: run_eval.sh hardcodes ROOT=<absolute path>


LIMITATIONS
------------
- full_streaming KV cache grows linearly (~4.3 GB for 60-min video); manageable on A6000
- duo_plus_rekv is approximate (streaming heads use n_local, not Duo's recent_size)
- decord must use 1 thread on this cluster (avcodec thread safety issue)
- rekv positional encoding: retrieved blocks lose absolute position (distance scaling only)
- partial/OOM result JSONs show null aggregate_metrics; must delete and rerun
- duo_streaming quality is lower than rekv at s=0.5; tune --sparsity lower to improve


KEY PARAMETERS
---------------
--method           required   full_streaming | duo_streaming | rekv | duo_plus_rekv
--dataset          rvs_ego    rvs_ego | rvs_movie
--sample-fps       0.5        frames/sec to sample (must match across compared runs)
--max-new-tokens   64         tokens to generate per answer
--n-local          15000      ReKV GPU local window (tokens)
--retrieve-size    64         ReKV top-k blocks from CPU at question time
--n-frame-tokens   196        visual tokens per frame (LLaVA-OV = 196, must match model)
--attn-dir         see scripts path to Duo trained head weights (.tsv files)
--sparsity         0.5        fraction of streaming heads (0=all full, 1=all streaming)
--video-decode-threads 1      keep at 1 (threading crashes on this cluster)
--max-videos       all        limit videos (use 1 for smoke test)
--num-chunks / --chunk-index  sharding for multi-GPU runs
--resume                      resume from partial result JSON


BACKEND REQUIREMENTS
---------------------
flash-attn          full-attention forward (strongly recommended)
block_sparse_attn   Duo sparse streaming (required for paper-faithful Duo)
flashinfer          RMSNorm/RoPE acceleration (optional; torch fallback)
decord              fast video decoding (recommended; imageio fallback)
triton              ReKV inner attention kernel (optional; --rekv-fattn flag)
