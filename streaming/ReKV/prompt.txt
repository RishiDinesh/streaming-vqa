Streaming ReKV handoff
======================

Use this file as the fast handoff context for the streaming `ReKV` lane.

Active scope
------------
- Focus on `streaming/ReKV`.
- Treat `duo_attn/` as support code for comparison methods.
- Main active datasets:
  - `RVS-Ego`
  - `RVS-Movie`
- Main active methods:
  - `full_streaming`
  - `duo_streaming`
  - `rekv`

- Main detailed guide:
  - `streaming/ReKV/ReKV_Duo.md`

Current branch intent
---------------------
- Current working branch: `exp/nv-gpu-inference`
- Preferred execution target: NVIDIA SLURM cluster
- Preferred Python environment for new runs: `conda activate duo`
- Preferred runtime validator:
  - `python -m streaming.ReKV.validate_runtime_env`
- Historical note:
  - the original project/papers/repos are NVIDIA-first
  - `exp/amd-gpu-inference` is the porting/adaptation branch

Streaming contract
------------------
- causal frame ingest only
- one sampled frame per forward pass
- shared state across questions from the same video
- no offline whole-video prefill
- question-time visibility uses sampled timestamps strictly earlier than each conversation `end_time`
- dataset loader currently sorts conversations by `end_time`
- all comparison methods must use the same sampled-frame schedule

Current corrected setup
-----------------------
- `run_eval.py` writes a shared `evaluation_manifest`.
- The manifest now records question ordering correctly as:
  - `dataset_loader_sorted_by_end_time`
- `run_eval.py` now supports sharded evaluation with:
  - `--num-chunks`
  - `--chunk-index`
- Feature-cache reuse is guarded by schedule-equivalence checks.
- All methods share the same prompt-prefill / decode path in `methods.py`.
- Duo methods expose actual backend resolution, not just library availability.
- Duo methods support:
  - `--duo-strict-no-sdpa-fallback`
- ReKV retrieval on short probe horizons no longer crashes when `n_init` exceeds the currently available remainder.

Backend policy
--------------
- Use `validate_runtime_env` before trusting any Duo comparison.
- On NVIDIA:
  - expect `accelerator_backend = cuda`
  - only treat Duo as native sparse streaming if `streaming_attn_backend_actual = blocksparse`
  - prefer native CUDA libraries and kernels rather than AMD fallback assumptions
- On ROCm:
  - historical audit support remains
  - `validate_rocm_env` is a compatibility alias to `validate_runtime_env`
  - ROCm-specific notes live in `streaming/ReKV/ROCM_Backend_Audit.md`

Working interpretation
----------------------
- `rekv` remains the safest mainline method to trust and extend.
- `duo_plus_rekv` is explicitly an approximate hybrid:
  - ReKV retrieval remains native
  - Duo routing is applied over the assembled ReKV answer context
- Do not describe Duo fallback runs as equivalent to native block-sparse Duo runs.

Suggested workflow
------------------
1. Validate runtime:
   `python -m streaming.ReKV.validate_runtime_env`
2. Optionally precompute feature cache:
   `python -m streaming.ReKV.precompute_features ...`
3. Run eval:
   `python -m streaming.ReKV.run_eval ...`
   or `sbatch streaming/ReKV/run_eval.sh ...`
   or `NUM_CHUNKS=N sbatch --array=0-(N-1) scripts/run_streaming_eval_slurm_array.sh`
4. Judge:
   `python -m streaming.ReKV.judge_results --in-place ...`
5. Plot:
   `python -m streaming.ReKV.plot_results ...`
6. Profile:
   `python -m streaming.ReKV.profile_streaming ...`
   or `sbatch streaming/ReKV/profile_streaming.sh ...`

Current result story
--------------------
- Retained full-eval trees still support the conservative conclusion:
  - `rekv` is the main method to trust
  - `duo_plus_rekv` is useful to compare and tune, but not a clear default winner
- The branch is now better instrumented for backend-specific interpretation, especially for Duo.
- The branch should be kept aligned with the official NVIDIA-first ReKV and Duo implementations:
  - ReKV paper/repo: https://arxiv.org/abs/2503.00540 , https://github.com/Becomebright/ReKV
  - Duo paper/repo: https://arxiv.org/abs/2410.10819 , https://github.com/mit-han-lab/duo-attention
