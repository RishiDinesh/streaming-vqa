Streaming semantics
-------------------
- causal frame ingest only
- one sampled frame per forward pass
- shared state across questions from the same video
- no offline whole-video prefill
- question-time visibility uses sampled timestamps strictly earlier than each conversation `end_time`
- all comparison methods must use the same sampled-frame schedule and same cutoff policy

Current corrected comparison setup
----------------------------------
- The four active comparison methods are:
  - `full_streaming`
  - `duo_streaming`
  - `rekv`
  - `duo_plus_rekv`
- `run_eval.py` now writes a shared `evaluation_manifest` so runs expose:
  - shared streaming protocol
  - method family
  - runtime backend path
  - Duo deploy settings
  - retrieval/offload semantics
- resume now fails if comparison-critical config drifts

Important implementation facts
------------------------------
- All methods now share the same prompt-prefill / answer flow in `methods.py`.
- ReKV methods no longer double-apply the question prompt.
- ReKV answer-time assembly now force-includes the true local recent window.
- `duo_plus_rekv` is explicitly an approximate hybrid:
  - ReKV retrieval stays native
  - Duo is applied over the assembled ReKV answer context
  - the hybrid is not the official standalone Duo tuple-cache algorithm
- ReKV and hybrid result JSONs now report:
  - retrieved block counts
  - retrieval latency
  - forced-local-window diagnostics
  - assembled context layout stats
- On MI300X, the ROCm-specific ReKV decode path needed a fix:
  - clone `last_hidden_state` before LM-head projection

Current Duo deployment note
---------------------------
- The audited streaming Duo runs used training-aligned deploy settings:
  - `sink=512`
  - `recent=1024`
- This replaced the older reduced `sink=256` / `recent=512` assumption for fairness checks.
- The offline-only `256/512` setting may still be useful for separate experiments, but it is not the default truth for streaming comparisons.

Workflow
--------
1. Validate or build the shared feature cache if needed.
2. Run evals through `python -m streaming.ReKV.run_eval` or the launcher scripts.
3. Judge with `python -m streaming.ReKV.judge_results --in-place`.
4. Plot with `python -m streaming.ReKV.plot_results`.
5. Profile representative videos with `python -m streaming.ReKV.profile_streaming`.

Current takeaway
----------------
- The flow is in a much better correctness state now than before the audit.
- Current subset evidence says:
  - `rekv` and `duo_plus_rekv` are now being compared fairly
